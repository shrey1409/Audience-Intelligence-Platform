## L-001: Pydantic-settings constructor kwargs override env vars

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Symptom:** Setting `DATABASE__SCHEMA=test_schema` as a process env var had no effect on `settings.database.schema`, which always returned `"public"`.

**Root Cause:** When `Settings(**merged_yaml_dict)` is called, the kwargs passed to the constructor have higher priority than env vars in Pydantic-settings. The YAML value `schema: public` is passed as `database={"schema": "public", ...}` to `Settings()`, and Pydantic-settings treats constructor kwargs as the highest priority source.

**Fix:** Added `_apply_env_overrides(merged_dict)` that iterates `os.environ` and injects `DATABASE__*` env vars into the corresponding sections of the merged dict before calling `Settings(**merged)`.

**Pattern to use:**
```python
# In Settings.from_yaml_and_env():
merged = _load_and_merge_yaml()
_apply_env_overrides(merged)  # env vars win over YAML
return cls(**merged)
```

**Also:** JSON-parse values in `_apply_env_overrides()` because env vars are always strings but Pydantic fields may expect `list[str]`. `API__API_KEYS='["key"]'` needs `json.loads()` to become `["key"]`.

## L-002: Alembic autogenerate hardcodes schema literal

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Symptom:** Integration tests with custom schema names created tables in `public` instead of the test schema, even when `DATABASE__SCHEMA` was correctly set.

**Root Cause:** `alembic revision --autogenerate` captures the current value of `settings.database.schema` (which was `"public"`) and writes `schema="public"` as a string literal in every `op.create_table()` call. This hardcoded value ignores any future schema configuration.

**Fix:** After autogenerate, add `_schema = settings.database.schema` at the top of the migration file and replace all `schema="public"` with `schema=_schema`.

**Rule:** Always review autogenerated migrations before committing. Never blindly accept them.

## L-003: Settings singleton cannot be overridden by module reload in tests

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Symptom:** `importlib.reload(app.core.config)` after setting `os.environ['DATABASE__SCHEMA'] = 'test'` did not change the schema used by running code.

**Root Cause:** The `settings` singleton is created at module import time. `importlib.reload()` creates a new module object with a new `settings` instance, but all existing code that already ran `from app.core.config import settings` holds a reference to the OLD object. The reload creates an orphan object that no one uses.

**Fix:** Run schema-sensitive code in a **subprocess**. A subprocess has a fresh Python interpreter where env vars are in effect before any import. The `DATABASE__SCHEMA` env var set in `subprocess.env` will be picked up by the first import of `app.core.config` in that process.

**Pattern:**
```python
import subprocess, sys, os
env = {**os.environ, "DATABASE__SCHEMA": test_schema}
result = subprocess.run(
    [sys.executable, "-W", "ignore::UserWarning", "scripts/run_migrations.py"],
    env=env, capture_output=True, text=True,
)
```

## L-004: `Timestamp` is not a valid SQLAlchemy type

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Symptom:** `ImportError: cannot import name 'Timestamp' from 'sqlalchemy'` when writing ORM models.

**Root Cause:** Mistakenly typed `Timestamp` instead of `DateTime`. `sqlalchemy.Timestamp` does not exist. The correct import is `from sqlalchemy import DateTime`.

**Rule:** All timestamp columns use `DateTime` from `sqlalchemy`, not `Timestamp`, not `TIMESTAMP`.

## L-005: `from decimal import Decimal` required in every model with Numeric columns

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Symptom:** `NameError: name 'Decimal' is not defined` when running `mapped_column(Numeric(10,2), default=Decimal("0"))`.

**Root Cause:** `Decimal` is not a built-in; it must be explicitly imported from the `decimal` standard library module.

**Rule:** Any ORM model that uses `Numeric(...)` with a Python-side default must include `from decimal import Decimal`.

## L-006: SQLAlchemy column `default=` vs `server_default=`

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Key distinction:**
- `default=uuid.uuid4` — Python-side default, computed when the ORM object is instantiated in Python. The UUID is available before the INSERT hits the database. Good for PKs because it enables unit testing without a DB connection.
- `server_default=func.now()` — PostgreSQL-side default, set by the database server on INSERT. Not available in Python until after the row is committed and refreshed. Good for `created_at` (authoritative server timestamp).

**Pattern for PKs:** Always `default=uuid.uuid4` (Python-side)
**Pattern for timestamps:** `server_default=func.now()` for `created_at`; `server_default=func.now(), onupdate=func.now()` for `updated_at`

## L-007: Feature store upsert requires PK as conflict target

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Key fact:** `INSERT ... ON CONFLICT (user_id) DO UPDATE` works because `user_id` is the PRIMARY KEY, which implies UNIQUE. No additional `UNIQUE` constraint needed. The conflict target must name the PK column (not an expression).

## L-008: Pre-commit hook execution order and `E501` debugging

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Key fact:** When multiple hooks fail, `black` runs before `flake8`. If `black` auto-formats a file, flake8 then checks the formatted file. The sequence: black → isort → flake8. If black reformats something and flake8 still fails, the remaining flake8 errors are real (not black-fixable).

**Common E501 sources:**
- Module-level docstring on line 1 (can't use `# noqa` on docstrings — must shorten)
- Class docstrings that are one long sentence
- Long column names in CHECK constraints
- String literals in test assertions

**Fix pattern:** Shorten the literal text, not add `# noqa: E501` on docstrings.

## L-009: ga4_identity_bridge design rationale

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Key insight:** The bridge table grows **incrementally** — one row per resolved `user_pseudo_id`. Without it, each ETL run must re-scan the entire `ga4_events` table to rebuild the `user_pseudo_id → user_id` mapping. With the bridge, only NEW events since the last run need to be processed for new resolutions.

At 15M events (synthetic), the bridge might have 60K rows. Scanning 60K is O(60K). Scanning 15M is O(15M). At NYPost scale (500M events), this is the difference between a 30-second identity step and a 3-hour one.

## L-010: Docker init scripts only run on empty volume

**Date:** 2026-05-31
**Phase:** 2 — Database Schema
**Key fact:** The SQL files in `docker/postgres/init/` run ONLY when the `postgres_data` Docker volume is created for the first time (empty container). If the volume already exists with data, the init scripts are skipped.

**Clean slate procedure:**
```bash
docker compose down -v   # -v removes volumes
docker compose up -d postgres
# init scripts will run on the fresh volume
```

## L-011: `ultraplan` requires `master-specification.md` (hyphenated)

**Date:** 2026-05-31
**Phase:** 2 — Planning
**Symptom:** ultraplan failed with "File does not exist: .claude/specs/master-specification.md" because the file was saved as `masterspecification.md` (no hyphen).

**Fix:** Created `master-specification.md` as an alias. Both files must exist in `.claude/specs/`. The ultraplan prompt hardcodes the hyphenated filename.

**Rule:** When creating spec files, always check what filename the downstream commands expect before naming the file.
